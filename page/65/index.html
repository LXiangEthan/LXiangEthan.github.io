MyBatis 之所以能够允许我们“随便填写”接口方法的返回值，并在运行时正确地处理这些返回值，背后是一系列复杂但有序的机制和约定。下面我将详细解释 MyBatis 是如何实现这一点的。

1. 接口与映射文件的绑定

在 MyBatis 中，一个接口通常对应一个映射文件（XML 或注解形式）。这个映射文件描述了接口中每个方法对应的 SQL 语句，以及这些 SQL 语句执行后应如何映射到 Java 对象。

2. 动态代理与调用链

当 MyBatis 初始化时，它会扫描你的接口，并为每个接口创建一个动态代理对象。这个代理对象会在运行时拦截接口方法的调用，并根据映射文件中的配置执行相应的 SQL 语句。

3. SQL 执行与结果处理

当代理对象拦截到一个方法调用时，它会根据映射文件中的配置找到对应的 SQL 语句，并执行这个 SQL 语句。执行完成后，MyBatis 会根据 SQL 语句的类型（查询、更新、删除等）以及映射文件中的配置来处理结果。

4. 返回值类型匹配与映射

对于查询操作，MyBatis 会根据映射文件中配置的 resultType 或 resultMap 来确定如何映射查询结果到 Java 对象。如果返回类型是自定义的 Java 类型或复杂类型（如列表、Map 等），MyBatis 会使用反射机制来创建这些对象，并将查询结果的字段值填充到对象的属性中。

对于更新、删除或插入操作，MyBatis 通常会将受影响的行数作为返回值。这个返回值类型通常是 int 或 long，与接口方法的返回类型相匹配。

5. 类型处理器与自定义映射

MyBatis 还提供了类型处理器（Type Handler）机制，允许你自定义如何将 Java 类型与数据库中的类型进行映射。这给了你更大的灵活性，可以处理一些特殊类型或复杂的映射需求。

总结

通过接口与映射文件的绑定、动态代理与调用链、SQL 执行与结果处理、返回值类型匹配与映射以及类型处理器与自定义映射等一系列机制，MyBatis 实现了对接口方法返回值的灵活处理。这使得我们可以“随便填写”接口方法的返回值，只要确保映射文件中的配置与这些返回值相匹配即可。

需要注意的是，虽然 MyBatis 提供了很大的灵活性，但我们在使用时仍需遵循一定的规范和约定，以确保代码的可读性、可维护性和性能。****

使用if标签可以实现动态sql，if标签使用来完成条件判断的，如果满足条件那么if标签里面的内容就会显示，否则就没有，if标签里面写test属性，就是写条件的地方

但是这里还有问题，不知道大家有没有发现，那就是如果我们第一个name参数如果没有怎么办?
我们会发现，程序报错了，不难看出来，因为and的存在，如果没有第一个name参数，那么where后面就会直接跟着一个and，导致sql语句报错

解决方式：
笨办法，在where后面加上一个恒等式1=1，代表一定满足，这样即使后面多出来一个and程序也不会报错

聪明办法：使用where标签

使用where标签可以自动删除语句前面的and字段，而且也不用我们自己写where

在mybatis中我们使用#{}来表示占位符
对于这个占位符，mybatis会自动把我们定义的接口实现的传入的参数也传入这个占位符里面

这个地方我自己发现了个问题，那就是mybatis是如何实现我们自己定义的接口，无论什么返回值，比如我们使用set或者map返回值，mybatis都会装入数据，但是我们返回多个数据只用一个Bean类存放就会报错，这边我猜测的是mybatis官方可能使用了多条判断，利用反射获取到了我们的方法的返回值如果是List等容器数据类型，那么就会把返回的数据放入容器中，但是如果返回的只是一个Bean类，那么就会直接返回一个结果，对于这个问题，文心一言的解释可以看看

MyBatis里面参数占位符:
#{}会把sql语句里面替换为❓，可以防止sql注入
${}就不会自动替换，无法防止sql注入，但是这个也是有点用的就和js里面的模板字符串是一样的，可以实现动态查询
对于这个参数占位符也可以设置参数类型

但是对于这个值parameterype也可以不写的

这边还有一个问题，那就是我们需要在sql中写<，但是xml本身数据格式不允许我们书写<，这时候我就想起了之前学习xml时候解决这个问题的方法，请看看xml这一章,里面介绍了用转义符号&lt代替<或者使用来解决问题
